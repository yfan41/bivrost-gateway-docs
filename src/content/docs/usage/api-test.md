本页面提供部分读取/写入数据的接口，用做实际部署前的验证测试。本页所有接口详细说明可参考《通讯协议》2.5.1. 直接读写下对应接口说明。

在接口测试前，如有修改过**机台配置**、**机组配置**、**任务配置**、或**通讯配置**等相关设置，请先在主页点击**重启服务**。

![接口测试页面](/img/manual/api-test/api-test-page.png)

在上方"选择机台"栏选择在**机台配置**页面激活的机台，在接口目录中选择要测试的 HTTP 接口（默认为常用数据），点击**监控**/**发送**按键，可以获取到该机台对应接口的数据。

![选择机台](/img/manual/api-test/machine-picker.png)

![选择接口](/img/manual/api-test/interface-picker.png)

## 5.8.1. 常用数据接口 {#common-data}

监控常用数据，定期（约 5 秒）请求一系列常用接口包括机台状态，加工程序信息，当前刀号，加工计数，警报信息等接口，刷新数据如下图。点击**停止监控**退出监控。

![常用数据监控](/img/manual/api-test/common-data.png)

## 5.8.2. 读取数据接口 {#read-data}

除了常用数据接口，写入 PLC 数据，写入刀补数据以外的接口都是读取数据接口，发送读取数据接口请求，以加工计数接口为例，得到数据如下图。加工计数接口返回的是机床加工计数结果。下方以 `/api/cnc/` 开头的为 HTTP 接口地址。`{}` 中为结果，其中 "count" 为名称，2806 为名称对应数值。接口详细介绍，可参考《通讯协议》2.5.1. 直接读写下对应接口说明。

![读取加工计数](/img/manual/api-test/read-count.png)

```
/api/cnc/ReadCount?MachineID=2118

{
  "count": 2806
}
```

部分接口支持多通道设备，可以获取指定通道号的数据。默认为通道 0，即主通道。只有一个通道的设备保持默认通道 0 即可。如下图中，通过机台状态接口获取指定通道的状态。

![选择通道](/img/manual/api-test/channel-picker.png)

对于读取 PLC 数据接口，需要补充区域，起始地址，数据数量，数据类型等输入量，可以参考 [6.2.1. PLC 数据任务](/reference/command-format/#plc-task)中的说明，以及《通讯协议》2.5.1.18. readPlcData 读取 PLC 数据。

![读取 PLC 数据](/img/manual/api-test/read-plc-data.png)

对于读取刀补数据接口，需要刀补号等输入量，详见《通讯协议》2.5.1.12. readOffsetData 读取刀补数据。

![读取刀补数据](/img/manual/api-test/read-offset-data.png)

对于读取刀具寿命与读取刀具寿命详情接口，根据机台系统型号不同需要补充不同的变量。此例子中，系统为 Fanuc 发那科，需要补充刀具的组号与组内序号，详见《通讯协议》2.5.1.21. readToolLife 读取刀具寿命。

![读取刀具寿命](/img/manual/api-test/read-toollife.png)

## 5.8.3. 写入数据接口 {#write-data}

写入数据接口包括写入 PLC 数据，写入刀补数据接口。这两个接口除了类似读取 PLC 数据接口，与读取刀补数据需要补充区域，起始地址，数据类型，刀补号等输入量，还需要在请求体补充写入数据，详见《通讯协议》2.5.1.19. writePlcData 写入 PLC 数据，2.5.1.14. writeOffsetData 写入刀补数据。点击**格式化**按键将请求体优化为标准格式。

写入 PLC 数据请求体示例：

```json
{
  "data": [
    1,
    2,
    3
  ]
}
```

![写入 PLC 数据](/img/manual/api-test/write-plc-data.png)

写入刀补数据请求体示例：

```json
{
  "lengthWear": 0.0011,
  "radiusWear": 0.13,
  "lengthGeom": 0.1,
  "radiusGeom": 0.012
}
```

![写入刀补数据](/img/manual/api-test/write-offset-data.png)

写入数据成功，返回错误码（errorCode，0 代表成功）与错误内容（errorMsg），详见《通讯协议》2.2. 错误处理。

```json
{
  "errorCode": 0,
  "errorMsg": "Success"
}
```

## 5.8.4. 测试失败排查 {#troubleshooting}

接口测试失败回复"接口调用失败"，如下图，在返回结果中给出错误代码，错误信息，错误详情，与故障排查建议。

![接口调用失败](/img/manual/api-test/call-failed.png)

如接口测试失败，请检查：

1. **机台配置**页面中的设置；
2. **机台配置**/**任务配置**页面改动后是否在主页点击**重启服务**；
3. 机床上的网络设置（详见《机床设置》）是否正确，部分机床设置后需要断电三秒重启以生效；
4. 网线连接口，交换机等硬件是否连接正常；
5. 机床 IP 地址是否已被占用。当局域网中同时接入办公电脑等其它设备时，可能出现机床的 IP 被占用，导致机床与网关通信失败。这时候可以尝试给机床换个未被占用的 IP。注：一个 IP 地址 ping 不通，不代表这个 IP 地址没被占用，有些设备出于安全考虑禁用了 ping 回复。
6. 另一种 IP 被占用的情况是网段被占用。常见于采集网络与其它内网接入同一个交换机时，采集网络网段与另一内网网段相同。当此问题出现时，将冲突的内网设备网段或者采集网络网段改到另一未被使用的网段。
